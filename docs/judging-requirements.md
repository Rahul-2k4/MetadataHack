## Official Judging Criteria

Judges evaluate submissions on:

1. Potential Impact
2. Creativity and Innovation
3. Technical Excellence
4. Best Use of OpenMetadata
5. User Experience
6. Presentation Quality

## What This Means for Submission Content

To score well against those criteria, this repo should always contain:

- A clear problem statement and target users
- Explicit OpenMetadata integration points (APIs/entities/features)
- Runnable setup and reproducible demo path
- Visible evidence of engineering quality (structure, clarity, reliability)
- A concise demo flow and polished README

## Disqualification-Critical Rules (Must Follow)

- Coding/design should start only after official hackathon start.
- If using AI tools, disclosure is required in submission materials.
- Rule or code-of-conduct violations may lead to disqualification.

## Team Submission Gate (Internal)

Before final submission, confirm all of these are true:

- Demo script passes end-to-end without manual patching.
- README aligns with what is shown in demo.
- Architecture diagram matches implemented flow.
- Final video/demo link is ready.
- Project stays inside hackathon scope.
